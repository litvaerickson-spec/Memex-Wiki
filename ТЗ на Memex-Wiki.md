# **ТЗ: Реализация гибридной системы локальной памяти «Memex-Wiki» через MCP-сервер для Antigravity 2.0**

**Цель проекта:** Создать отказоустойчивую, автономную и полностью приватную систему долгосрочной памяти для ИИ-агентов. Система должна накапливать знания в виде человекочитаемого графа (Markdown-файлов с YAML-метаданными) и позволять малой локальной модели эффективно извлекать, фильтровать и обновлять информацию последовательными порциями, исключая зависание локального железа и перегрузку контекстного окна.

## **1\. Концептуальный стек и методы работы**

Архитектура системы разделена на два независимых слоя: **Слой хранения** (накопление знаний) и **Вычислительный слой** (навигация и выполнение кода). Она базируется на следующих методах и технологиях:

### **Слой накопления знаний (Логика хранения)**

* **Инкрементальная компиляция знаний (Knowledge Compilation):** Данные обрабатываются и структурируются ИИ-агентом *в момент их добавления* в систему. Новый документ из папки raw/ за один проход атомизируется на сущности и интегрируется в связанные страницы wiki/.  
* **Двухэтапный синтаксический анализ (Two-Step Ingest):** Агент сначала генерирует скрытую цепочку рассуждений (Chain-of-Thought) для сопоставления сущностей и их синонимов, и только вторым вызовом аккуратно обновляет файлы Markdown, предотвращая засорение базы.  
* **Автоматический линтинг графа (Graph Linting):** Периодический фоновый аудит структуры для устранения логических противоречий, обновления индексов и поиска «сиротских» страниц.  
* **Версионирование через Git:** Любое изменение памяти со стороны ИИ фиксируется в локальном Git-репозитории, позволяя осуществлять откат (git rollback) при ошибках рефлексии.

### **Вычислительный слой (Логика выполнения)**

* **Контекст как частично наблюдаемая среда (Context-as-Environment):** Запрещается передавать сырой текст документов напрямую в промпт модели. Текст изолируется в программной среде выполнения (Sandbox). Модель взаимодействует с ним исключительно через вызовы встроенных скриптов.  
* **Рекурсивная декомпозиция и последовательный конвейерный опрос (Sequential Batching):** *Внимание: замена параллельных запросов для стабильности на локальном железе.* Текст нарезается на мелкие чанки, которые обрабатываются локальной LLM в строго последовательном однопоточном режиме, что гарантирует защиту от нехватки видеопамяти (VRAM OOM).  
* **Диффузионная запись ответа (State-Machine Answer):** Ответ формируется внутри REPL-переменной как конечный автомат. Модель может циклически корректировать выводы, пока не выставит программный флаг готовности.

## **2\. Архитектура данных (Storage Layer)**

Вся память физически хранится в локальной директории (стандартный Obsidian Vault). Агент обязан строго следовать следующей иерархии папок:

Plaintext  
memex\_vault/  
├── raw/                      \# Исходные документы (неизменяемые файлы: pdf, txt, логи)   
└── wiki/                     \# Скомпилированные Markdown-страницы, управляемые ИИ   
    ├── system/               \# Системный профиль и контекст пользователя  
    │   ├── user\_profile.md   \# Жесткие факты о пользователе (философия, стек, ТЗ)  
    │   └── system\_rules.md   \# Архитектурные правила и запреты для ИИ  
    ├── index.md              \# Оглавление базы знаний и краткие описания страниц   
    ├── log.md                \# Append-only лог всех операций компиляции памяти \[cite: 12\]  
    ├── \[concept-name\].md     \# Страницы концептов, проектов, сущностей (в нижнем регистре) \[cite: 12\]  
    └── \[source-summary\].md   \# Страницы-выжимки конкретных источников из raw/ \[cite: 12\]

### **Стандарт формата страниц (wiki/\*.md) с YAML-метаданными**

Каждый файл в папке wiki/ обязан строго начинаться с YAML-блока метаданных (Frontmatter) для обеспечения стабильной работы Python-парсеров.

Markdown  
\---  
id: concept-name  
type: concept / project / summary  
tags: \[ai, local, tech\]  
last*\_updated: ГГГГ-ММ-ДД*  
*relations: "\[\[related-concept-1\]\], \[\[related-concept-2\]\]"*  
*\---*  
*\# Название Страницы*

*\[cite\_*start\]**\*\*Summary\*\***: Одно-два предложения, описывающие суть данной страницы. \[cite: 12\]

\[cite*\_start\]**\*\*Sources\*\***: Список исходных файлов из папки \`raw/\`, на которых основана страница. \[cite: 13\]*

*\---*

*\#\# Основное содержание*  
*\[cite\_*start\]Текст разбит на короткие абзацы со строгими заголовками. \[cite: 14\]  
\[cite*\_start\]Все упоминания связанных сущностей оформляются как двунаправленные вики-ссылки: \[\[related-concept\]\]. \[cite: 15\]*  
*\[cite\_*start\]Каждое фактологическое утверждение должно содержать явную цитату (source: filename.ext). \[cite: 15\]

\#\# Related pages  
\- \[cite\_start\]\[\[related-concept-1\]\] \[cite: 16\]  
\- \[cite\_start\]\[\[related-concept-2\]\] \[cite: 16\]

## **3\. Вычислительный движок (RLM Engine)**

Для обеспечения стабильности на малых локальных моделях ИИ-агент **не должен** генерировать и выполнять произвольный Python-код на лету. Вместо этого в модуле memex\_tools.py реализуется жестко закодированный набор функций, оптимизированный под алгоритмы RLM.

### **Алгоритм RLM-фильтрации длинного контекста (Последовательный конвейер)**

Если объем целевого файла (или группы файлов) превышает лимит оперативного контекста локальной модели (константа MAX\_CHUNK\_TOKENS \= 3000) , движок выполняет строго последовательный цикл:

\[Тяжелый файл знаний\] ➔ (Нарезка на чанки по 3000 токенов)  
                                 │  
                                 ▼  
                     ┌───────────────────────┐  
                     │ Опрос Чанка №1 (LLM)  │ ➔ Ответ: "НЕТ" (Пропуск)  
                     └───────────────────────┘  
                                 │  
                                 ▼  
                     ┌───────────────────────┐  
                     │ Опрос Чанка №2 (LLM)  │ ➔ Ответ: "ДА" ➔ \[Сохранить ключевую строку\]  
                     └───────────────────────┘  
                                 │  
                                 ▼  
\[Финальный сжатый промпт из строк "ДА"\] ➔ (Локальная LLM) ➔ \[Точный ответ пользователю\]

1. **Chunking:** Скрипт нарезает текстовый файл на перекрывающиеся куски по MAX\_CHUNK\_TOKENS.  
2. **Sequential Stream Query:** Скрипт отправляет к локальной LLM (через Ollama/vLLM) последовательную очередь запросов (строго по одному, без параллелизма): «Есть ли в данном фрагменте ответ на вопрос \[Вопрос\]? Ответь строго ДА или НЕТ и выпиши ключевую строку».  
3. **Synthesis:** Скрипт отсекает куски со значением НЕТ, собирает релевантные строки из ответов ДА и формирует из них один концентрированный финальный промпт для модели.

## **4\. Спецификация MCP-сервера для Antigravity 2.0**

ИИ-агент должен реализовать MCP-сервер на Python с использованием официального SDK (mcp). Взаимодействие со средой выполнения Antigravity 2.0 осуществляется строго через **транспорт стандартного ввода-вывода (**stdio**)**.  
Сервер регистрирует и предоставляет наружу следующие инструменты (Tools):

### **ingest\_source**

* **Описание:** Запускает процесс обработки нового файла, добавленного в папку raw/.  
* **Аргументы:** filename (string).  
* **Логика:** Читает файл ➔ Выделяет сущности ➔ Проверяет наличие дубликатов в wiki/ ➔ Создает/обновляет страницы по шаблону с YAML-массивом ➔ Прописывает \[\[вики-ссылки\]\] ➔ Добавляет записи в wiki/index.md и wiki/log.md ➔ Делает git commit.

### **query\_memory**

* **Описание:** Точечный поиск по базе знаний без перегрузки промпта.  
* **Аргументы:** query (string).  
* **Логика:** Сначала парсит YAML-метаданные в wiki/system/user\_profile.md и wiki/index.md ➔ Локализует список потенциально релевантных файлов ➔ Применяет алгоритм последовательной RLM-фильтрации (из Раздела 3\) к этим файлам ➔ Возвращает в Antigravity 2.0 очищенный контекст.

### **lint\_memory**

* **Описание:** Аудит целостности графа знаний и метаданных.  
* **Аргументы:** отсутствуют.  
* **Логика:** Сканирует wiki/ ➔ Проверяет валидность YAML-структур ➔ Находит изолированные страницы (orphan pages) ➔ Находит ссылки \[\[ссылка\]\] без физического файла ➔ Ищет прямые противоречия в фактах (сверяя даты last\_updated) ➔ Возвращает отчет.

## **5\. Пошаговый воркфлоу для реализации агентом**

Разработчик (ИИ-агент) должен строго по шагам написать и развернуть систему:

* **Шаг 1: Инициализация окружения и Git.** Создать структуру папок memex\_vault/, выполнить git init, сгенерировать файлы конфигурации, дефолтный wiki/index.md, wiki/log.md и структуру системных профилей в wiki/system/.  
* **Шаг 2: Модуль RLM-инструментов (**memex\_tools.py**).** Написать функции чтения/разметки Markdown. Реализовать отказоустойчивый парсер YAML-фронтматера и регулярных выражений для поиска двунаправленных ссылок вида \\\[\\\[(.\*?)\\\]\\\]. Написать функцию безопасной последовательной нарезки и опроса чанков через локальное API.  
* **Шаг 3: Реализация бизнес-логики.** Интегрировать в промпты компиляции жесткое требование: любое утверждение должно сопровождаться тегом (source: имя\_файла.ext). Заблокировать на уровне файловой системы (или Python-кода) любую возможность записи или модификации файлов внутри папки raw/. Принудительно переводить имена создаваемых файлов в нижний регистр через дефисы.  
* **Шаг 4: Развертывание MCP-интерфейса (**mcp\_server.py**).** Импортировать mcp.server, обернуть функции из memex\_tools.py в инструменты MCP, настроить запуск сервера через транспорт stdio.

## **6\. Набор автоматических тестов для проверки (Валидация)**

Система считается успешно реализованной, если внутренний тестовый скрипт успешно проходит следующие этапы в полностью автономном локальном режиме:

1. **Тест инжеста:** Положить в raw/ файл astra-spec.txt с текстом *"Почтовый клиент для Astra Linux должен работать без административных прав"* и вызвать инструмент ingest\_source.  
   * *Ожидаемый результат:* В папке wiki/ должен появиться файл astra-spec.md с валидными YAML-метаданными, а также создаться/обновиться концептуальный файл astra-linux.md со ссылкой на первый. В лог и индекс внесены записи.  
2. **Тест RLM-поиска:** Вызвать инструмент query\_memory с запросом *"Какие права нужны для почтового клиента в Astra Linux?"*.  
   * *Ожидаемый результат:* Модель должна вернуть ответ: *"Для работы почтового клиента права администратора не требуются (source: astra-spec.txt)"*. При этом размер контекста, отправленного в локальную LLM за один итерационный шаг, не должен превышать лимит MAX\_CHUNK\_TOKENS. Локальная Ollama не должна уходить в зависание или перезагрузку по OOM.  
3. **Тест Git-фиксации:** После выполнения инжеста скрипт проверки должен выполнить команду git status и подтвердить автоматическое создание коммита для измененных файлов папки wiki/.
