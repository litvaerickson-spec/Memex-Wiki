Предложенная гибридная концепция (Компиляция по Карпатому \+ Выполнение по RLM) разделяет систему памяти на два независимых слоя.  
Ниже приведена подробная карта методов (алгоритмических подходов) и технологий (конкретного софта и библиотек), которые составляют фундамент этой архитектуры.

## **1\. Методы (Алгоритмические подходы и парадигмы)**

Методы описывают саму логику обработки данных — то, *как* ИИ думает, структурирует знания и выполняет вычисления.

### **Слой накопления знаний (по Карпатому)**

* **Инкрементальная компиляция знаний (Knowledge Compilation):** Вместо классического динамического RAG (когда ИИ ищет куски документов прямо во время вашего вопроса), система компилирует данные *в момент их добавления*. Новый документ из папки raw/ прочитывается один раз, сжимается, атомизируется на сущности и навсегда «впаивается» в связанные страницы wiki/.  
* **Двухэтапный синтаксический анализ (Two-Step Ingest):** Агент не пишет страницу сразу. Сначала он генерирует скрытую цепочку рассуждений (Chain-of-Thought), извлекая сущности, сопоставляя их с синонимами (алиасами), и только вторым вызовом аккуратно обновляет файлы Markdown.  
* **Автоматический линтинг графа (Graph Linting):** Периодический фоновый метод (например, раз в сутки), при котором агент сканирует базу на наличие логических ошибок: находит «сиротские» страницы (без ссылок на них), разрешает фактологические противоречия и обновляет корневой index.md.

### **Слой вычислений и навигации (по MIT RLM)**

* **Контекст как частично наблюдаемая среда (Context-as-Environment):** Парадигма, запрещающая передавать текст документов внутрь промпта. Текст загружается как изолированная строковая переменная (P) в среду выполнения (Sandbox). Модель видит только метаданные и структуру, общаясь с контекстом программно.  
* **Рекурсивная декомпозиция и параллельное делегирование (Recursive Task Delegation):** Метод борьбы с «забыванием информации в середине промпта». Корневой агент пишет Python-код, разбивает массив данных на логические фрагменты и вызывает параллельный пакет дочерних моделей (llm\_batch()), отдавая каждой строго один фрагмент текста для точечного извлечения ответа.  
* **Диффузионная запись ответа (State-Machine Answer):** Ответ формируется внутри REPL-переменной answer \= {"content": "", "ready": False}. Агент может дописывать, стирать и корректировать свои выводы в течение нескольких итераций кода, пока не выставит флаг ready: True.

## **2\. Технологии (Программный стек и инструменты)**

Технологии — это конкретные инструменты, библиотеки и форматы файлов, на которых пишется система.

       \[ ТЕХНОЛОГИИ \]                                \[ МЕТОДЫ \]  
┌──────────────────────────┐               ┌──────────────────────────┐  
│  Obsidian (MD \+ YAML)    │ ────────────\> │  Компиляция знаний       │  
├──────────────────────────┤               ├──────────────────────────┤  
│  Git / Git Hooks         │ ────────────\> │  Линтинг и Версионирование│  
├──────────────────────────┤               ├──────────────────────────┤  
│  Python REPL / Sandbox   │ ────────────\> │  Контекст как среда      │  
├──────────────────────────┤               ├──────────────────────────┤  
│  Official \`rlms\` Library │ ────────────\> │  Рекурсивное делегирование│  
└──────────────────────────┘               └──────────────────────────┘

### **Хранилище (Storage & Interface)**

* **Obsidian (Markdown \+ YAML Frontmatter):** Файловая база данных. Двунаправленные ссылки вида \[\[Имя\_Страницы\]\] выступают в роли ребер графа знаний. Блоки YAML в начале файлов используются для хранения жестких метаданных (теги, даты, связи, авторы), которые легко считывать кодом.  
* **Git:** Используется как движок контроля версий памяти. Любое изменение, внесенное агентом в ваши файлы, оставляет след. Вы всегда можете через git diff посмотреть, какие именно факты агент переписал в вашей памяти, и сделать git rollback в случае его ошибки.

### **Вычислительное ядро (Execution Backend)**

* **Python 3.11+ REPL / Изолированные песочницы (Docker / WASM):** Защищенная программная среда, где Root-модель выполняет свои скрипты через exec(). Внутри доступны стандартные библиотеки для быстрого текстового поиска (os, pathlib, re, json).  
* **Официальная библиотека rlms:** Официальный открытый фреймворк от MIT OASYS (pip install rlms), предоставляющий готовый интерфейс для подмены стандартного вызова llm.completion() на рекурсивный rlm.completion().  
* **MCP (Model Context Protocol):** Протокол от Anthropic, используемый для того, чтобы локально запущенный агент (например, Claude Code или кастомный скрипт) мог безопасно читать, искать и модифицировать файлы в вашей локальной папке Obsidian через стандартные вызовы API.

### **Модели и Инфраструктура ИИ**

* **Локальные движки (Ollama / vLLM / Llama.cpp):** Инфраструктура для запуска моделей без интернета.  
* **Эффективные open-weights модели (например, Qwen 2.5 / Qwen 3.5 или Llama 3):** За счет того, что RLM-скрипты берут на себя всю грязную работу по поиску и фильтрации текста с помощью регулярных выражений, вам не нужны гигантские коммерческие облачные модели. Локальная модель среднего размера (14B–32B параметров) в такой связке показывает точность выше, чем топовая закрытая модель, которой просто «в лоб» скормили огромный файл.

## **Как это работает в связке (Пример одного запроса)**

| Шаг | Что происходит за кулисами | Какие технологии работают |
| :---- | :---- | :---- |
| **1\. Запрос** | Вы спрашиваете: *«Какие у нас дедлайны по интеграции Whisper в Astra Linux?»* | Интерфейс ввода |
| **2\. Поиск по графу** | Root-модель пишет Python-скрипт, который прочесывает YAML-шапки в Obsidian, находит файлы с тегом \#AstraLinux и выписывает все \[\[ссылки\]\] из них. | **Python REPL \+ Obsidian** |
| **3\. Фильтрация данных** | Скрипт находит нужные абзацы через обычный re.search(), не загружая тонны лишнего текста в память ИИ. | **Библиотека rlms** |
| **4\. Сборка ответа** | Дочерний микро-агент анализирует очищенные строчки, выписывает даты и аккуратно складывает их в переменную ответа. | **Ollama / Локальная LLM** |
| **5\. Фиксация в памяти** | Система обновляет лог log.md в Obsidian, связывая ваш вопрос и новый сгенерированный инсайт. | **Git \+ MCP** |
